## Section Guidelines

### Section 1: Accuracy Metrics
- Calculate from ALL feedback files
- Percentages should add up to 100% (valid + FP + won't fix)
- Average accuracy is mean of all "overall_accuracy" values from JSONs

### Section 2: False Positive Analysis
- Group findings by pattern_category from JSON
- Only show pattern categories that have at least 1 finding
- Flag categories with FP rate > 30% as "high FP rate"
- Include developer comments as context for high-FP patterns
- Pattern categories reflect repository-specific bugfix patterns from `context/historical-bugfix-patterns.md`. Use the correct pattern table for the analyzed repository.

### Section 3: Per-Report Breakdown
- One row per feedback JSON file
- Sort by date (newest first)
- Include risk level from original report

### Section 4: Prevention Suggestions
- Only generate if there are patterns with FP rate > 30%
- OR if the same false positive reason appears across 2+ reports
- Be specific and actionable (reference real developer comments)

### Section 5: Trends
- Only calculate if 3+ reports available
- Group by month if data spans multiple months
- Show direction: up = improving, down = declining, stable = +/-0.5 variation

## Feedback JSON Schema

Each code review generates a feedback JSON file stored in `reports/feedback/`:

```json
{
  "ticket_id": "HM-XXXXX",
  "repository": "HealthBridge-Web",
  "review_date": "YYYY-MM-DD",
  "risk_level": "High/Medium/Low",
  "overall_accuracy": 8.5,
  "findings": [
    {
      "id": 1,
      "title": "Finding title",
      "pattern_category": "Edge Cases",
      "severity": "Critical/High/Medium/Low",
      "verdict": "valid/false_positive/wont_fix",
      "developer_comment": "Optional developer explanation"
    }
  ]
}
```

### Verdict Definitions

| Verdict | Meaning | Impact on Accuracy |
|---------|---------|-------------------|
| **valid** | Finding is correct and actionable | Increases accuracy score |
| **false_positive** | Finding is incorrect - code is actually safe | Decreases accuracy score |
| **wont_fix** | Finding is technically correct but won't be addressed | Neutral (not counted against accuracy) |

### Accuracy Score Calculation

```
Accuracy = (valid_count / (valid_count + false_positive_count)) * 10

Note: "won't fix" findings are excluded from accuracy calculation
as they represent correct findings that are intentionally deferred.
```

## Interactive Feedback Flow

The feedback process is integrated into the code review workflow:

1. **Code review completes** - Agent generates report with findings
2. **Developer reviews findings** - Each finding is presented with 4 options:
   - **Valid** - Finding is correct, will be fixed
   - **False Positive** - Finding is wrong, code is safe
   - **Won't Fix** - Finding is correct but won't address now
   - **Provide More Information** - Need deeper analysis before deciding
3. **Deep analysis (optional)** - If "Provide More Information" selected, agent reads actual code, searches for sibling patterns, and generates detailed findings file
4. **Feedback saved** - JSON file saved to `reports/feedback/[TICKET]-feedback.json`
5. **Accuracy report** - Aggregated from all feedback JSONs on demand

## Generating the Accuracy Report

**Command:**
```
Generate accuracy report from feedback data
```

**Process:**
1. Scan `reports/feedback/` for all `*-feedback.json` files
2. Parse each JSON file
3. Aggregate metrics across all reports
4. Generate accuracy report using the template above
5. Save to `reports/feedback/accuracy-report.md`

**When to generate:**
- After accumulating 3+ feedback files (minimum for meaningful analysis)
- At the end of a sprint or release cycle
- When investigating code review quality trends
